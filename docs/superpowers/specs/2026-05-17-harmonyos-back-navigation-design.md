# HarmonyOS Shell 返回键行为增量设计

## 背景

- 当前 `WebPage` 仅处理页面顶部返回按钮，且行为偏向 Web 内后退。
- 用户反馈两个问题同时存在：
  - 即使输入 `https://baidu.com/` 也无法正常打开，说明当前 Web 体验仍不稳定。
  - “切换服务器 -> 管理服务器”不可点，说明 Web 与 ArkUI 浮层的交互关系需要继续收敛。
- 用户新增明确要求：
  - 如果网页还没加载出来，按返回键要先停止加载，再退回到初始输入网址的界面。
  - 该规则同时覆盖系统返回键与顶部左上角返回按钮。

## 目标

- 统一系统返回键与顶部返回按钮的行为。
- 在加载中优先停止加载并返回 `ConnectPage`。
- 保留已加载完成页面的 Web 内后退能力。
- 不影响现有服务器存储与最近一次 URL 恢复逻辑。

## 非目标

- 本次不处理双击返回退出应用。
- 本次不重做服务器管理页交互。
- 本次不引入新的路由栈管理方案。

## 推荐方案

采用单一入口 `handleBackAction()` 统一处理所有返回动作。

原因：

- 用户看到的“返回”只有一个概念，系统键和顶部按钮不应出现两套规则。
- Web 正在加载、浮层打开、错误态展示这三类状态本身已具备明确优先级，集中处理最稳定。
- 便于后续继续接入更多状态，例如下载确认、文件选择器等。

## 返回优先级

统一从高到低执行：

1. 若服务器切换浮层已打开：关闭浮层，不离开页面。
2. 若当前处于错误态：直接回到 `ConnectPage`。
3. 若当前页面仍在加载：停止加载，再回到 `ConnectPage`。
4. 若页面已加载完成且 Web 存在可回退历史：执行 Web 内后退。
5. 否则：回到 `ConnectPage`。

## 交互细节

### 顶部返回按钮

- 不再直接调用 WebView 的后退接口。
- 改为调用统一的 `handleBackAction()`。

### 系统返回键

- 接入住系统返回键处理。
- 系统返回键与顶部返回按钮完全复用同一套逻辑。

### 加载中返回

- 调用 ArkWeb 控制器的停止加载能力。
- 停止加载后立即切回 `ConnectPage`。
- 不要求等待网页触发 `onPageEnd` 再跳转。

### 回到输入页后的体验

- `ConnectPage` 仍保留最近一次输入/保存的服务器信息。
- 用户回到输入页后能继续修改 URL、重试连接，或直接点击已保存服务器。

## 状态与数据影响

- 不新增持久化字段。
- 继续复用：
  - `last_url`
  - `last_server_id`
  - 服务器列表
- 返回到 `ConnectPage` 只改变当前页面，不清空历史存储。

## 风险与处理

### ArkWeb 停止加载接口兼容性

- 若 `stop()` 或等价接口在当前 API 下名称不同，需要按现有编译环境修正。
- 若无法可靠判断“已停止”，仍以用户可见效果优先：立即跳回 `ConnectPage`。

### 系统返回键接入方式

- 若当前 ArkUI 页面组件不直接提供声明式返回处理，需要通过页面生命周期或 Ability 侧回调接入。
- 实现时以“系统返回键可统一进入 `handleBackAction()`”为验收目标，不限定内部写法。

## 验收标准

- 点击顶部返回按钮时，行为与系统返回键一致。
- 服务器切换浮层打开时，任一返回键优先关闭浮层。
- 页面错误态展示时，任一返回键直接回到 `ConnectPage`。
- 页面仍在加载时，任一返回键都会停止加载并回到 `ConnectPage`。
- 页面加载完成且存在浏览历史时，任一返回键优先执行 Web 内后退。
- 页面加载完成但无浏览历史时，任一返回键回到 `ConnectPage`。
