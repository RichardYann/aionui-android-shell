# AionUi Android Release Copy Refresh (Design)

## Goal

Refresh the Android shell release presentation so that:

- GitHub Release titles and release notes use a polished bilingual Chinese and English format
- the repository README homepage better communicates the product's current capabilities and highlights
- the release experience feels more intentional, attractive, and product-facing without changing the already working release automation flow

## Confirmed Decisions

- Keep the current release automation workflow structure and update the copy inside it
- Use a fixed bilingual release title and release body template
- Upgrade the repository README into a more product-showcase oriented homepage
- README should clearly describe current functionality and product highlights
- Emoji are allowed and can be used in moderation to improve tone and visual rhythm

## Scope

### In Scope

- Update `.github/workflows/android-release.yml` release name text
- Replace auto-generated-only release notes with a curated bilingual release body template
- Update the repository root `README.md` homepage structure and copy
- Add a more polished presentation of:
  - what the app is
  - what it can do now
  - what makes it distinctive
  - where to download releases

### Out of Scope

- No change to Android application runtime behavior
- No change to release build mechanics unless required for copy templating
- No redesign of `android-shell/README.md` unless needed for consistency later
- No formal signing work in this copy refresh task

## User Experience

### GitHub Release Page

Each release should look like a proper product release instead of a plain auto-generated changelog page.

The release page should:

- have a bilingual title
- present a short bilingual overview of the app
- show practical install and download information
- clearly state the current release-stage limitation around formal signing
- still include a changelog section, but in a cleaner framed way

### Repository Homepage

The repository README should feel closer to a lightweight product landing page:

- clear title and positioning
- concise bilingual intro
- readable feature blocks
- visually pleasant section ordering
- release and download entry visible without digging

The tone should be clean, confident, and a little lively, not noisy or overly decorative.

## Recommended Copy Direction

### Option Chosen

Use a bilingual product-showcase style with moderate emoji support.

Reasons:

- the app is already usable and now benefits from a stronger presentation layer
- bilingual release notes help both Chinese-speaking and English-speaking readers
- a polished README improves first impression and download discoverability
- limited emoji use can make the page feel more modern and approachable without becoming messy

## Technical Design

### Release Title

Replace the current release name pattern:

- from: `v0.1.0`
- to: `AionUi Shell for Android / AionUi Android 客户端 v0.1.0`

This pattern should be generated from the workflow version input so future releases remain consistent.

### Release Body Template

Replace the current auto-generated-only release notes with a fixed bilingual markdown template written directly in the release workflow.

The release body should have these sections:

1. `Overview / 版本说明`
2. `Highlights / 本版亮点`
3. `Downloads / 下载内容`
4. `Install / 安装方式`
5. `Current Limitations / 当前限制`
6. `Changelog / 更新记录`

#### Intended Content Shape

`Overview / 版本说明`

- one short English sentence explaining this is the Android shell for AionUi WebUI
- one matching Chinese sentence explaining the same thing

`Highlights / 本版亮点`

- mention current available capabilities in a user-facing way
- expected items:
  - server connection and quick reopen
  - immersive top navigation behavior
  - loading feedback during connect/refresh
  - file upload and system download support
  - AionUI launcher branding

`Downloads / 下载内容`

- explicitly name the release APK asset
- clarify that the release asset is attached on the page

`Install / 安装方式`

- state that installation is done through direct APK sideload
- note that Android may require allowing installation from unknown sources

`Current Limitations / 当前限制`

- explicitly state that this stage prioritizes release automation
- note that formal production signing will be added later

`Changelog / 更新记录`

- keep a concise link to compare or full changelog
- do not rely on the current default autogenerated body alone

### README Homepage Refresh

The root `README.md` should be reorganized into a more presentation-focused structure.

Recommended section order:

1. Title
2. Short bilingual intro
3. `✨ Features / 功能`
4. `🌊 Highlights / 特点`
5. `📦 Release / 发布下载`
6. `🛠️ Build / 构建`

#### README Content Expectations

`Title`

- keep the project name clear and recognizable

`Short bilingual intro`

- explain that this is a lightweight Android shell for AionUi WebUI
- keep it short and welcoming

`✨ Features / 功能`

- describe what the app currently supports in practical terms
- expected bullets:
  - first-run server URL entry
  - later auto-reopen of last server
  - back, refresh, and change server controls
  - upload support for file input
  - download support through Android system download manager

`🌊 Highlights / 特点`

- frame the product qualities rather than just checklist features
- expected themes:
  - lightweight and focused
  - immersive navigation behavior
  - clear loading feedback
  - simple self-hosted access flow
  - AionUI visual branding

`📦 Release / 发布下载`

- include the GitHub Releases page entry
- mention the versioned APK naming pattern
- note current release-stage limitation around formal signing

`🛠️ Build / 构建`

- keep essential local development notes and CI references
- move technical build details later in the page so the top reads more like a polished homepage

## Copy Style Rules

- Use bilingual headings where appropriate
- Keep sentences short and readable
- Use emoji sparingly to create rhythm, not decoration overload
- Prefer product-facing wording over internal implementation jargon
- Avoid inflated marketing language
- Preserve technical accuracy

## Files To Modify

- Modify: `.github/workflows/android-release.yml`
- Modify: `README.md`

## Acceptance Criteria

- GitHub Release title becomes bilingual and versioned
- GitHub Release body uses a stable bilingual markdown template
- The release page clearly communicates download, installation, and current limitations
- Root README homepage is rewritten in a more polished product-showcase style
- README clearly presents current functionality and distinguishing characteristics
- Emoji use feels tasteful and improves readability instead of distracting from content

## Risks And Mitigations

- Risk: release notes become too long
  - Mitigation: keep sections short and scannable

- Risk: README becomes too promotional and loses clarity
  - Mitigation: balance showcase language with concrete feature bullets

- Risk: copy diverges from actual product state
  - Mitigation: only mention features already implemented and verified
